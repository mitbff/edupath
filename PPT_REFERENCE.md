# EduPath 智能导学 Agent PPT 参考文档

本文档用于给制作 PPT 的同学快速理解项目内容，重点说明“项目讲什么、功能怎么串、技术路线怎么表达、答辩时怎么演示”。

## 1. 项目一句话介绍

EduPath 是一个面向智慧教育场景的智能导学系统，通过“在线答题 → 错因诊断 → 知识追踪 → 知识图谱可视化 → 个性化推荐 → 教师干预”的闭环，帮助学生发现薄弱知识点，也帮助教师进行班级学情分析和分层教学。

推荐题目：

```text
EduPath 智能导学 Agent：基于知识图谱与学习诊断的个性化推荐系统
```

更正式一点：

```text
面向智慧教育的知识图谱驱动学情诊断与个性化学习路径推荐系统设计与实现
```

## 2. 项目背景与痛点

可以从三个痛点展开：

- 学生只知道“错了几题”，不知道为什么错。
- 教师只看到班级平均分，难以及时发现具体薄弱知识点和高频错因。
- 传统题库系统更多是刷题，缺少诊断、追踪、推荐和教学干预闭环。

本项目的解决思路：

- 每道题绑定知识点、难度和错因标签。
- 系统根据答题情况更新知识点掌握度。
- 用 ECharts 知识图谱展示学习路径和薄弱节点。
- 根据薄弱点推荐微课、例题、巩固题和错题复盘任务。
- 教师端生成班级热力图、错因排行、学生分层和下一节课教学建议。

## 3. 核心功能闭环

PPT 中建议画成一张流程图：

```text
学生在线答题
    ↓
题目级错因诊断
    ↓
知识点掌握度更新
    ↓
知识图谱节点变色
    ↓
个性化学习建议与练习生成
    ↓
教师端班级诊断与分层作业
```

讲述重点：

- 系统不是只展示页面，而是把学生作答数据转化为诊断结果。
- 诊断结果会同时影响学生端报告和教师端看板。
- 同一套诊断引擎可以适配数学、英语、Python 编程等不同课程场景。

## 4. 主要功能模块

### 4.1 在线答题模块

学生完成 5-10 道诊断题，系统记录：

- 学生答案
- 正确率
- 答题用时
- 每题对应知识点
- 每题难度
- 每题典型错因

PPT 展示建议：

- 放“智能答题”页面截图。
- 标注每道题背后有知识点、难度、错因标签，不是普通选择题。

### 4.2 错因诊断模块

系统根据错误题目输出错因解释，例如：

- 概念不清
- 公式误用
- 审题偏差
- 计算失误
- 方法选择不当
- 未作答

讲述重点：

- 错因不是学生手动填写，而是由题目元数据和选项解释自动匹配。
- 诊断结果可以追溯到具体题目和具体知识点。

### 4.3 知识图谱模块

系统将知识点表示为图结构，例如数学导数场景：

```text
函数基础 → 极限思想 → 导数定义 → 单调性 → 极值应用 → 导数应用
```

节点颜色含义：

- 绿色：已掌握
- 黄色：待巩固
- 红色：薄弱

PPT 展示建议：

- 放知识图谱截图，这是最抓眼的一页。
- 重点讲“薄弱节点会根据诊断结果自动变色”。

### 4.4 知识追踪模块

系统使用简化版知识追踪算法更新掌握度：

```text
新掌握度 = 旧掌握度 × 0.7 + 本次表现 × 0.3
```

本次表现分综合考虑：

```text
表现分 = 正确率 × 0.5 + 难度系数 × 0.2 + 答题稳定性 × 0.3
```

讲述重点：

- 不是简单按正确率判断掌握度。
- 题目难度和答题稳定性也会影响诊断结果。
- 这种设计体现了“学习过程追踪”。

### 4.5 个性化推荐模块

系统根据最薄弱知识点推荐：

- 微课
- 例题
- 巩固题
- 错题复盘任务
- 综合迁移任务

讲述重点：

- 推荐不是随机资源，而是围绕薄弱知识点生成。
- 学生看到的是下一步学习路径，而不是单纯分数。

### 4.6 AI 学习建议模块

当前系统提供“AI 增强”开关：

- 关闭时：使用本地规则模板生成建议，保证稳定演示。
- 开启时：后端尝试调用 OpenAI-compatible 大模型接口。
- 未配置 API Key 或调用失败时：自动降级为规则模板。

推荐答辩说法：

```text
本系统不是直接把题目丢给大模型判断，而是先通过知识点、错因标签、答题耗时和掌握度模型完成结构化诊断，再将诊断结果交给大模型生成更自然的个性化学习建议。这样既保证了诊断过程可解释，也提升了反馈内容的自然度。
```

讲述重点：

- 核心诊断由系统算法完成。
- 大模型负责语言生成和建议润色。
- 这种“规则诊断 + 大模型表达增强”的混合架构更可靠。

### 4.7 教师端班级诊断看板

教师端展示：

- 班级知识点热力图
- 高频错因排行
- 学生分层名单
- 学习风险预警
- 下一节课教学建议
- 自动生成分层作业

PPT 展示建议：

- 放教师看板截图。
- 重点讲系统同时服务学生端和教师端，体现精准教学。

## 5. 技术架构

PPT 中建议画成三层架构：

```text
前端展示层
HTML / CSS / JavaScript / ECharts
    ↓
后端服务层
Python FastAPI / REST API / AI 生成服务
    ↓
数据持久层
SQLite / 题库 / 知识点 / 作答日志 / 掌握度快照
```

### 前端技术

- HTML / CSS / JavaScript：实现页面、导航、答题交互和诊断报告。
- ECharts：实现知识图谱、雷达图、热力图、柱状图等可视化。
- Fetch API：请求后端接口，获取题库并提交诊断数据。

### 后端技术

- FastAPI：提供 REST API。
- Pydantic：校验前端提交的数据结构。
- Uvicorn：运行后端服务。
- CORS：支持本地前端访问后端接口。
- AI 服务层：实现 AI 开关、大模型调用和规则降级。

### 数据库技术

- SQLite：轻量级数据库，适合课程大作业演示和本地部署。
- 保存内容包括场景、知识点、图谱边、题库、选项、资源、作答日志、诊断记录和掌握度快照。

## 6. 后端接口设计

主要接口：

```text
GET  /api/health
GET  /api/ai/status
GET  /api/bootstrap
GET  /api/scenarios
GET  /api/questions/{scenario_id}
POST /api/diagnosis
GET  /api/dashboard/{scenario_id}
```

重点接口：

- `/api/bootstrap`：前端启动时加载完整场景配置。
- `/api/diagnosis`：提交学生答案，返回正确率、错因统计、掌握度更新、推荐资源和 AI 报告。
- `/api/ai/status`：返回后端是否配置大模型 API Key。
- `/api/dashboard/{scenario_id}`：返回教师端看板数据。

PPT 讲述重点：

- 项目不是纯静态页面，后端有真实接口。
- 诊断数据会写入 SQLite。
- FastAPI 自带接口文档，可以通过 `/docs` 展示。

## 7. 数据库设计

主要数据表：

```text
scenarios              课程场景
knowledge_nodes        知识点节点
knowledge_edges        知识点依赖关系
questions              题库
question_options       题目选项与选项解释
resources              推荐资源
diagnoses              诊断记录
answer_logs            作答日志
mastery_snapshots      掌握度快照
```

讲述重点：

- 题目、知识点、错因、资源之间是结构化关联的。
- 每次诊断都会产生记录，可以用于后续学习历史分析。
- 数据库设计支持继续扩展更多课程场景。

## 8. 多场景扩展设计

当前支持三个场景：

- 高一数学：导数单元
- 高中英语：阅读理解
- Python 编程：入门项目

多场景的意义：

- 说明系统不是只为某一套题写死。
- 不同课程共用同一套诊断流程。
- 只需要替换题库、知识图谱和资源库，就能扩展到更多课程。

## 9. 创新点总结

可以在 PPT 中作为单独一页：

- 从“只给分数”升级为“错因解释 + 学习路径 + 教师干预”。
- 从单一 demo 升级为多课程场景可配置平台。
- 用知识图谱表达知识点依赖关系。
- 用简化知识追踪算法动态更新掌握度。
- 用 ECharts 将薄弱知识点和班级学情可视化。
- 用证据链解释诊断来源，降低 AI 黑箱感。
- 采用“规则诊断 + 可选大模型表达增强”的混合 AI 架构。
- 同时覆盖学生端和教师端，形成智慧教育闭环。

## 10. PPT 推荐结构

建议做 10-12 页：

1. 封面：项目名称、成员、课程信息。
2. 背景与痛点：为什么需要智能导学系统。
3. 系统目标：要解决什么问题。
4. 功能闭环：在线答题到教师干预的完整流程。
5. 学生端功能：答题、错因诊断、学习建议。
6. 知识图谱与知识追踪：算法和可视化展示。
7. 教师端看板：热力图、错因排行、学生分层。
8. 技术架构：前端、后端、数据库、AI 服务层。
9. 核心算法与数据流：掌握度更新、诊断接口、数据写入。
10. AI 增强设计：规则模板、大模型开关、自动降级。
11. 系统演示流程：如何现场展示。
12. 创新点与总结展望。

## 11. 演示流程

推荐现场演示：

1. 打开系统首页，展示整体界面。
2. 切换数学、英语、Python 场景，说明多场景可配置。
3. 进入“智能答题”，点击“填入演示答案”。
4. 提交诊断，展示学生报告。
5. 展示知识图谱节点颜色变化。
6. 展示 AI 学习建议和推荐资源。
7. 打开“教师看板”，展示班级热力图、错因排行和分层作业。
8. 进入“技术实现”，展示数据来源、后端同步和 AI 生成层状态。
9. 打开 `http://127.0.0.1:8000/docs` 展示 FastAPI 接口文档。

## 12. 答辩常见问题准备

### Q1：这个系统真的用了后端吗？

答：

```text
用了。前端启动时会请求 FastAPI 的 /api/bootstrap 获取场景、题库和知识图谱；提交答题后会调用 /api/diagnosis，后端计算诊断结果并写入 SQLite 数据库。
```

### Q2：AI 是真的大模型吗？

答：

```text
系统设计了可插拔 AI 生成层。默认使用规则模板保证稳定演示；如果配置 API Key 并打开 AI 增强开关，后端会调用 OpenAI-compatible 大模型接口生成学习建议。如果调用失败，会自动降级为规则模板。
```

### Q3：为什么不用大模型直接判断学生错因？

答：

```text
因为教育诊断需要可解释性。本系统先用知识点、错因标签、难度和答题耗时形成结构化诊断，再让大模型负责生成自然语言建议。这样既可靠，也更容易解释诊断依据。
```

### Q4：项目如何扩展到其他课程？

答：

```text
系统将课程内容抽象为场景配置，包括知识点、知识图谱、题库、错因标签和资源库。扩展新课程时，只需要新增这些配置，诊断流程和可视化组件可以复用。
```

### Q5：这个系统和普通刷题网站有什么区别？

答：

```text
普通刷题网站主要记录对错，本系统会进一步分析错因、更新知识点掌握度、生成知识图谱状态、推荐学习资源，并为教师生成班级诊断和分层作业。
```

## 13. 可写进 PPT 的总结

```text
EduPath 将在线测评、知识追踪、知识图谱、个性化推荐和 AI 学习建议整合到一个完整闭环中。系统既能为学生提供具体可执行的学习路径，也能为教师提供班级学情诊断和分层教学依据，体现了智慧教育中“数据驱动诊断、算法辅助决策、人机协同教学”的设计思想。
```
